TT-NPE Installation Manual

This guide covers the clean install path for tt-npe without assuming you already have TT-XLA installed. The goal is simple: clone the repo, build it, set up the shell environment if you want, and verify the install with the bundled example workload.

Prerequisites

The validated tester run had these base tools available:

git
cmake
ninja
clang-17
python3
You do not need TT-XLA, a ResNet50 environment, or live hardware tracing just to install and smoke-test tt-npe.

Clone The Repository

git clone https://github.com/tenstorrent/tt-npe.git
cd tt-npe

If you use SSH with GitHub, cloning via git@github.com:tenstorrent/tt-npe.git is equally fine.

Build TT-NPE

./build-npe.sh

A successful build creates the install/ tree, including tt_npe.py, programmatic_workload_generation.py, and the shared libraries under install/lib/.

Optional Environment Setup

source ENV_SETUP

The shell helper is useful because it prepares the runtime environment and makes the CLI easier to launch. On the validated tester host, it emitted a non-fatal warning about optional package installation but still pointed to a working Python binary.

Verify The Install

source ENV_SETUP
tt_npe.py -w tt_npe/workload/example_wl.json

You can also run the CLI more explicitly:

python3 install/bin/tt_npe.py -w tt_npe/workload/example_wl.json

Success means the workload loads, the simulation completes, and cycle and bandwidth statistics are printed.

What This Install Path Does Not Assume

  • No TT-XLA checkout is required.
  • No TT-XLA wheel install is required.
  • No ResNet50 demo stack is required.
  • No live device profiling is required.

Common Pitfalls

Build fails early
Check the compiler toolchain, Python availability, and dependency downloads.
ENV_SETUP warns
Try the explicit Python invocation before treating the install as failed.
Expecting TT-XLA artifacts
tt-npe can run directly from its own workload JSON files.
